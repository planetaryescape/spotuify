## One daemon, four clients

The daemon embeds librespot and registers as the Spotify Connect device, so it is not remote-controlling some other player: it is the player. It also keeps a metadata cache and a local search index. Everything else is a view over one Unix socket.

<div class="surface-grid home-clients">
  <div>
    <h3>tui, the flagship</h3>
    <p>Ten screens: player, search, library, playlists, queue, history, devices, lyrics, diagnostics, notifications. Album art and a live spectrum rendered in the terminal. Synced lyrics. Vim-style keys. Quitting it changes nothing about the music.</p>

```bash
spotuify
```
  </div>
  <div>
    <h3>cli, the contract</h3>
    <p>If the TUI can do it, <code>spotuify &lt;command&gt;</code> can do it: 57 commands, output as table, json, jsonl, csv, or ids. fzf, jq, xargs, and your status bar are part of the product.</p>

```bash
spotuify search "luther vandross" --type track --format ids \
  | fzf \
  | xargs spotuify play-uri
```
  </div>
  <div>
    <h3>agents</h3>
    <p>Tell your agent what you are in the mood for. It runs the same CLI you do: plan candidates, resolve tracks, preview the playlist, create it once you approve. Prefer a server? The built-in MCP server exposes 37 tools, backed by the daemon that plays the audio.</p>

```bash
spotuify mcp
```
  </div>
  <div>
    <h3>macos menubar</h3>
    <p>A native SwiftUI player and menubar app on the same socket, for the moments you want a window instead of a terminal. One client among four, not the product.</p>
    <p><a href="/getting-started/install/#macos-app-dmg">Download the DMG</a></p>

</div>

d=$(mktemp -d)

spotuify resolve-tracks \

`spotuify` is for the part of your day where Spotify should behave like the rest of your tools. Open the TUI when you want to browse. Use commands when you already know the action. The one-command version is still there:

```bash
spotuify play "burial archangel"
```

<div class="surface-grid">
  <div>
    <h3>the music outlives the client</h3>
    <p>Playback, queue, devices, and cache live in the daemon. Close the TUI mid-song; nothing skips. Reattach from any client and the state is where you left it.</p>

```bash
spotuify status --format json
```
  </div>
  <div>
    <h3>metadata on disk, music streamed</h3>
    <p>The local store is metadata only: SQLite rows and a Tantivy index over your library, playlists, and listening history. Search and analytics answer from disk. The audio itself streams from Spotify, same as any Connect device.</p>

```bash
spotuify search "ambient" --source local --format jsonl
spotuify analytics top --kind artists --since 30d
```
  </div>
  <div>
    <h3>preview writes, then commit</h3>
    <p>Playlist writes can show the exact change before touching Spotify, and playlist and library writes land in an operation log you can reverse. Useful when a script or agent made the list.</p>

```bash
spotuify playlist create "Exile" --from candidates.jsonl --dry-run
spotuify playlist create "Exile" --from candidates.jsonl --yes
spotuify ops undo --dry-run
```
  </div>

## Recipes worth stealing

Start the TUI:

```bash
spotuify
```

Play the first search result:

```bash
spotuify search "never too much" \
  --type track --limit 1 \
  --format ids \
  | xargs spotuify play-uri
```

Feed your status bar from the daemon:

```bash
spotuify status --format json \
  | jq -r '
      if .item then
        .item.name + " - " + .item.subtitle
      else
        "not playing"
      end
    '
```

Find music you have not heard in a while:

```bash
spotuify analytics rediscovery --gap 90d --format json
```

Follow synced lyrics from the shell:

```bash
spotuify lyrics follow --lines 3
```

Ask an agent for a playlist without blind write access:

```bash
spotuify playlist plan "exile and returning home" --format json > plan.json
spotuify resolve-tracks --from plan.json --format jsonl > candidates.jsonl
spotuify playlist create "Exile and Return" --from candidates.jsonl --dry-run
```

## The bet

Plenty of Spotify terminal apps are built around the screen. `spotuify` is built around the command surface. If an action matters in the TUI, it should be reachable from `spotuify <command>` too.

That costs more command plumbing, but it gives us something concrete to test, script, and hand to agents. The screen is optional. The daemon and CLI are the contract.
