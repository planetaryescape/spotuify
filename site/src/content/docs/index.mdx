`spotuify` is for the part of your day where Spotify should behave like the rest of your tools. Open the TUI when you want to browse. Use commands when you already know the action. The one-command version is still there:

```bash
spotuify play "burial archangel"
```

The daemon keeps playback state. The TUI, CLI, scripts, and MCP server all talk to it.

## Recipes worth stealing

Start the TUI:

```bash
spotuify
```

Queue a small search set:

```bash
spotuify search "burial" --type track --limit 5 --format ids \
  | spotuify queue add --ids - --format json
```

Play the first search result:

```bash
spotuify search "never too much" \
  --type track --limit 1 \
  --format ids \
  | xargs spotuify play-uri
```

If you already use `fzf`, pick the result yourself:

```bash
spotuify search "luther vandross" --type track --format ids \
  | fzf \
  | xargs spotuify play-uri
```

Feed your status bar from the daemon:

```bash
spotuify status --format json \
  | jq -r '
      if .item then
        .item.name + " - " + .item.subtitle
      else
        "not playing"
      end
    '
```

Find music you have not heard in a while:

```bash
spotuify analytics rediscovery --gap 90d --format json
```

Ask an agent for a playlist without blind write access:

```bash
spotuify playlist plan "exile and returning home" --format json > plan.json
spotuify resolve-tracks --from plan.json --format jsonl > candidates.jsonl
spotuify playlist create "Exile and Return" --from candidates.jsonl --dry-run
```

## The bet

Plenty of Spotify terminal apps are built around the screen. `spotuify` is built around the command surface. If an action matters in the TUI, it should be reachable from `spotuify <command>` too.

That costs more command plumbing, but it gives us something concrete to test, script, and hand to agents. The screen is optional. The daemon and CLI are the contract.

```bash
spotuify search "ambient" --source local --format jsonl
spotuify cache status --format json
spotuify analytics top --kind artists --since 30d
```
