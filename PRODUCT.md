# Product

## Register

brand

## Users

Terminal-native Spotify users: engineers, Neovim users, shell-heavy makers, and coding agents that need a real music control surface. They are usually already in a terminal, often mid-task, and want to search, play, queue, inspect, or mutate Spotify without opening the desktop app.

## Product Purpose

`spotuify` is a daemon-backed, CLI-first Spotify controller and local music library runtime. Success means a user can run `spotuify`, play music, close the terminal, and keep listening. It also means every meaningful action has a pipeable command surface for scripts, tests, and agents.

## Brand Personality

Precise, nocturnal, stubborn.

The brand should feel like a serious terminal instrument, not a generic SaaS docs theme. It can look considered, but the visual weight should come from command fluency, reliable playback, and the feeling of a local tool that respects the user's hands.

## Anti-references

- Generic Starlight defaults with a swapped accent color.
- Purple-gradient developer landing pages.
- Card grids that could describe any CLI.
- Pretty TUIs that hide flaky playback.
- Spotify tools that are only a TUI, only a wrapper, or not scriptable.
- Copy that says "user-friendly", "powerful", or "clear insights" without a real trade-off.

## Design Principles

1. Show the command before the claim.
2. Make CLI-first obvious in the first viewport.
3. Treat the daemon as the system, not an implementation footnote.
4. Keep docs runnable: every guide section gives the reader a command.
5. Let the visual language feel terminal-native without becoming fake terminal cosplay.

## Accessibility & Inclusion

Aim for WCAG AA contrast, visible focus states, no motion required for comprehension, and readable line lengths. Support reduced-motion preferences. Do not rely on green alone for meaning.
